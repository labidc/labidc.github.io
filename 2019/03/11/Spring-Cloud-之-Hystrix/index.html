Spring Cloud 之 Hystrix

发表于 2019-03-11 | 更新于 2019-03-12 | 分类于 Spring Cloud

说到Spring Cloud就不得不想到 Netflix 公司, 因为该公司贡献出了微服务相关的很多开源项目。在这里我们今天来说说 Netflix 家族里的 Netflix ，相关官方文档. 我是一个使用任何一个中间件，都比较喜欢看官方文档的人, 因为官方的文档最权威

一、为什么用Hystrix？

先看看Hystrix有些什么功能

1. 隔离 : 隔离微服务调用下级服务过程中，下级服务故障或者响应能力不足造成整个系统崩溃。
2. 熔断 : 当失败率达到一定阈值时，熔断器触发停止继续调用下级服务，直接返回异常或者降级操作并响应给客户端， 防止请求资源堵塞，起到了限流的作用。
3. 降级 : 当下级服务出现 网络原因、服务响应太慢、异常、的时候调用的补偿函数，不直接触发异常返回给客户端。
4. 缓存 : 设置一定的缓存策略，返回缓存数据
5. 合并请求: 当调用下级服务的时候，下级多次相同调用开销太大，可以合并请求批量调用。
6. 实时监控、报警: 监控服务调用情况。

Hystrix 提供了哪些方式支持熔断器

1. 使用注解 @HystrixCommand 
2. 使用继承 HystrixCommands 和 HystrixObservableCommands 的实现
3. @FeignClient 用得最多的方式, 记得在配置文件设置 'feign.hystrix.enabled=true' 打开  FeignClient 对熔断器的支持

二、知道了Hystrix的作用，那我们就需要了解如何用好它。

配置文件的设置方式有两种

  1. 第一种：全局默认设置没有设置HystrixCommandKey的Hystrix方法
  1. 第二种：针对设置了自定义HystrixCommandKey的Hystrix方法，为该Hystrix方法专属使用

正式开始，我们这里以全局默认配置的范例来讲解。

该设置是熔断器的运行模式，有THREAD, SEMAPHORE两种，默认是THREAD模式，现在用Jmeter模拟每秒并发20个请求,看看在 THREAD, SEMAPHORE和模式下有什么 微服务A 里的线程有什么不同

首先 Tomcat的工作线程数量20

THREAD 模式下，为每个下级服务生成了多个线程hystrix-“微服务名称”-线程流水号与Hystrix-Timer-线程流水号

SEMAPHORE 模式下只生成 Hystrix-Timer-线程流水号 的线程

以上现象印证了官网配图

  1. 在THREAD模式下，新开了线程调用下级服务，这样可以更加真实的控制下级服务调用的“线程超时时间”。
    讲道理 必须满足“线程超时时间”>=“网络连接时间”+“请求响应时间”, 才能正常工作。如果你使用的@FeignClient，Feign默认使用的是ribbon, 尝试配置ribbon.ConnectTimeout 连接超时时间和 ribbon.ReadTimeout 响应超时时间 查看效果。ribbon 的超时会直接影响到Hystrix
  2. 在SEMAPHORE（信号量，多线程开发经验丰富的同学都已经很熟悉了）模式下，是使用的Tomcat的工作线程直接同步阻塞调用的下级服务。
    Hystrix-Timer-线程流水号 是用来管理上面两种模式的计时器。
  3. 总结：官网也是说了在 hystrixcommands 适合 THREAD 模式，当每秒数百个请求的时候，同时方法里调用下级服务又比较多的情况下是成倍数增长，这样是相当占用服务器资源，资源的占用有可能造成频繁的GC回收，以后有机会讲讲JVM相关原理，GC回收是需要重视的，而且线程太多，线程上下文环境的切换也是相当头痛的。SEMAPHORE 模式适合HystrixObservableCommands 调用，这样一个Tomcat工作线程进行批量处理工作。
注意看下图，现在我改一下上面的图，模拟下层服务的响应延迟(数据库IO并发) 小伙伴可以自己做实验
    graph LR
        start[用户请求某接口] --> input[微服务A,设置timeoutInMilliseconds=1000]
        input -- 调用微服务B -->  B{微服务B,线程延迟2秒}
        input -- 调用微服务C -->  C{微服务C,线程延迟2秒}
        input -- 调用微服务D -->  D{微服务D,线程延迟2秒}

该设置是线程超时时间设置，也就是由 “Hystrix-Timer-线程流水号” 来管理线程超时时间的

  1. 在THREAD模式下，更可控，也就是说当微服务A某方法调用下级的三个服务，如上图。
    timeoutInMilliseconds 设置1秒超时是小于下级的服务的2秒延迟的，三个下级服务都会超时，3个服务*1秒=3秒，服务A调用下级服务的函数
    感觉应该是3秒之后超时，但是由于线程模式是独立线程跑下级调用，可以理解可以是一种异步调用。我测试了每秒20个并发，最后平均服务A调用这三个服务的函数执行时间是1-3秒之间完成。
  2. 在SEMAPHORE下，就是另外一种现象，因为信号量的模式是使用Tomcat的工作线程去请求，按道理同步调用应该是 3个服务*2秒=6秒，我同样测试了每秒20个并发，实际效果是接近4-6秒。
  3. 总结：也就是说THREAD模式下线程的中断更可控，所以这就是拿空间换时间。不要小看这几秒的差距，当下级服务处理能力较差的情况下，可以大大提高吞吐量。

该设置是打开或者关闭线程超时的开关，如果这里填写false关闭之后，你就在服务A的Tomcat容器中看不到 Hystrix-Timer-线程流水号 相关线程了，
也就是上面的 timeoutInMilliseconds 设置直接无效了。

该设置默认值是true, 当timeoutInMilliseconds时间超时之后，直接中断该请求，该设置我尝试设置为false，实际上和true没发现有什么区别，感觉没有生效

该设置从官方文档来讲是当取消请求的时候中断线程，我尝试了各种实验方式都没有效果，于是尝试查询GitHub上的源码，想看看到底有什么操作。结果搜索到另外一个答案
https://github.com/Netflix/Hystrix/issues/1650
官方回答是该设置没有实现功能，只是为了想后兼容做铺垫整的一个配置。

该设置只在SEMAPHORE下有效，默认值是10，也就是最大能同时执行10个请求，当瞬间超过10个并发的时候，就返回调用降级操作了。

该设置官方文档说支持SEMAPHORE和THREAD模式，也就是说当大量降级操作的时候，降级操作是会占用资源的，当降级操作达到该设置的值，就直接返回异常给客户端了。我尝试把这个值设置成1，能查看到一种异常现象
com.netflix.hystrix.exception.HystrixRuntimeException: fallback execution rejected.

该设置是打开和关闭降级操作，默认true, 如果设置false，当下级服务调用出现超时、异常等情况就直接抛出异常到客户端了，不会做任何降级操作的调用。

下面开始才真正是熔断器配置的开始

既然是熔断器，这里才是真正的熔断器设置，熔断的含义就是当服务器请求达到一定负载之后（大量错误或超时请求），暂时性直接拒绝所有请求，这就是熔断的含义。高并发的情况下，可以有效防止雪崩，雪崩的含义就是服务本来正常情况下可以稳定吞吐一定量的并发，当大量并发的时候资源被占用，大量堵塞，反而吞吐量降低，甚至严重情况JVM崩溃。

这两个配置要一起讲

  1. requestVolumeThreshold 当并发在窗口时间内出现了requestVolumeThreshold次错误就开启熔断器, 这个所谓窗口时间没有看到有设置的地方，官方距离是10秒，不知道是否能改。
  2. errorThresholdPercentage 当错误率达到 errorThresholdPercentage 设置的值启动熔断，具体这个错误率是否是累计的也没有看到官方文档有说明

上面两个设置是强制打开熔断器和强制关闭熔断器

当断路器打开后，会直接拒绝请求，此时间是配置多长时候后再次尝试重新处理请求

hystrix 自带一个看板插件，metrics properties.主要用于统计配置执行情况 提供给看板插件使用，这里就不详细多讲了

就请求缓存和日志相关的配置，也不多讲了，小伙伴们可以自行百度或谷歌使用方法

在一些高并发的接口，你可以尝试合并请求的方式调用，该配置大家也可以自行百度或谷歌使用方法

Thread Pool Properties 线程池配置，以下配置看官方文档的介绍，应该只支持THREAD模式

官方文档介绍到其实当您正确配置时，hystrixcommand层的超时应该很少，但是如果不是网络延迟影响时间，或者在最坏的情况下，connect+read+retry+connect+read，注意这里提到了重试，我没有尝试实验该重试机制，大家知道有这么会事就好。

coreSize 该默认值是5，核心线程数也就管理 hystrix-“微服务名称”-线程流水号 的数量。
allowMaximumSizeToDivergeFromCoreSize 该设置表示是否开启弹性线程创建
maximumSize 当allowMaximumSizeToDivergeFromCoreSize 开启之后，maximumSize-coreSize=可以弹性创建的线程数
keepAliveTimeMinutes 当上面的弹性线程创建出来之后，挂起未运行的情况下 keepAliveTimeMinutes 分钟之后就回收掉

maxQueueSize 默认值是-1, 因此线程池的工作模式是 SynchronousQueue（同步队列） 永远保证队列中只有一个消费对象，生产者存一个，消费者取一下，如果两端处理能力差距太大，在hystrix这里如果出现堵塞就开始拒绝响应。 如果maxQueueSize设置大于0的任何值则线程池工作模式是LinkedBlockingQueue(阻塞队列), maxQueueSize设置5则表示阻塞队列里的队列长度是5。当高并发的时能保证至少5个请求能够在队列里处理，如果第6个并发请求到来，还没有一个队列空间释放出来的话，直接拒绝响应该请求。
queueSizeRejectionThreshold 拒绝队列大小，即使maxQueueSize的值没有达到，达到queueSizeRejectionThreshold该值后，请求也会被拒绝，该值主要是为了动态调整maxQueueSize的大小做的一个补偿方案。
以上提到拒绝处理，就是直接调用降级操作，不是直接返回异常。

前面提到了降级操作，如果FeignClient, 有两种降级操作的写法

  1. fallback 不关心产生降级操作的原因，统一处理。
  2. fallbackFactory 可以根据产生降级操作的异常，根据情况做不同的处理。
  3. 总结： 降级操作作为补偿性功能，其实可以做很多事情，比如返回友好错误响应给上级服务，根据不同的异常情况，写日志，或者一些补偿性数据库操作。

最后总结以下，Hystrix 该不该用完全看你项目的需要，如何用也是看你的项目需要，相关配置绝对不是设置越大越多就好，合适的设置才是正确的，后面有时间我会分享JVM一些原理来解释一下原因。（其实很多语言和架构的思想都是相通的）
